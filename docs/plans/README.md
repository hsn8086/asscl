# 计划文档规范

## 目的

计划文档用于：

- 明确复杂功能的实现路径
- 促进团队评审与共识
- 作为实现阶段的技术参考
- 保留决策上下文便于回溯

## 何时创建

满足以下任一条件时，应创建计划文档：

| 条件       | 阈值                |
| ---------- | ------------------- |
| 预估工作量 | > 4 小时            |
| 涉及模块数 | ≥ 2 个              |
| 技术风险   | 引入新依赖/架构变更 |
| 业务影响   | 需产品/设计确认     |

## 状态流转

```
┌─────────┐    评审通过    ┌─────────┐    完成开发    ┌─────────┐
│  draft  │ ───────────→ │ approved│ ───────────→ │  done   │
└─────────┘              └─────────┘              └─────────┘
     │                        │
     │ 评审不通过              │ 放弃/取消
     ↓                        ↓
┌─────────┐              ┌─────────┐
│ rejected│              │archived │
└─────────┘              └─────────┘
```

### 状态说明

| 状态       | 含义               | 可流转至           |
| ---------- | ------------------ | ------------------ |
| `draft`    | 草稿，正在编写     | approved, rejected |
| `approved` | 已批准，可开始实现 | done, archived     |
| `done`     | 已完成实现并验证   | archived           |
| `rejected` | 被拒绝，不再推进   | (终态)             |
| `archived` | 已归档             | (终态)             |

## 文件命名

```
NNNN-<slug>.md
```

- `NNNN`: 四位序号，全局递增
- `<slug>`: 英文短标识，小写中划线分隔

示例：`0003-offline-sync.md`

## 归档

状态为 `done` 或 `archived` 的计划，应在版本发布后移动至 `_archive/` 目录。
